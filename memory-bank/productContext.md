# Product Context: JBFH

## Зачем проект существует
Детские хоккейные школы в Беларуси нуждаются в централизованной системе для управления:
- Клубами и их филиалами
- Командами разных возрастов
- Тренерским составом
- Конструктором упражнений (лёд/земля) с классификацией по части тренировки, направленности, виду подготовки
- Инвентарём
- Тренировками, шаблонами тренировок, сетами упражнений (разминка/заминка)
- Программами обучения
- Детьми и их физическими показателями/нормативами

## Проблемы, которые решает
- Разрозненность данных о клубах и командах
- Отсутствие единой базы тренеров и их назначений
- Нет системы учёта инвентаря и упражнений
- Ручное управление доступом (решается ролевой моделью)
- Отсутствие расчёта параметров тренировочных упражнений (время отдыха, режим работы, общее время)
- Нет конструктора тренировок с шаблонами и сетами

## Как должна работать система
1. **Администратор** создаёт клубы, назначает пользователей
2. **Клуб** управляет своими командами, тренерами
3. **Методисты** (общие и клубные) просматривают данные
4. **Тренеры** видят свои команды, создают упражнения и тренировки
5. **Конструктор упражнений** — все могут создавать упражнения с классификацией (trainingPart, focuses, preparationType)
6. **Конструктор тренировок** — тренер наполняет тренировку упражнениями из каталога, система рассчитывает restDuration, workMode, totalTime
7. **Сеты** — готовые наборы упражнений для фаз тренировки (разминка/заминка)
8. **Шаблоны тренировок** — переиспользуемые заготовки, apply создаёт Training копированием

## User Experience цели
- Простой REST API с пагинацией на всех списках
- Swagger UI для ознакомления и тестирования
- Загрузка файлов через multipart (логотипы, картинки)
- Мягкое удаление (active=false) вместо физического для большинства сущностей
- Dynamic enum endpoints (GET /types, /training-parts, /focuses, /preparation-types) для фронтенда
- Калькуляционные ручки для предпросмотра расчётов без сохранения
- Фильтрация упражнений по всем классификаторам
